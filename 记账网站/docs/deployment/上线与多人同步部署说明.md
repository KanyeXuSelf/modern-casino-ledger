# 开局账本上线与多人同步部署说明

当前包装方式：

- 前端托管：GitHub Pages
- 共享数据：Supabase Postgres + Realtime
- 完整账本权限：Supabase Auth + `workspace_members`
- 单账单分享：`session_shares` + 带 token 的公开链接
- 冲突策略：最后一次保存覆盖前一次保存

## 1. 创建 Supabase 项目

1. 登录 [Supabase](https://supabase.com/)
2. 新建一个项目
3. 打开 SQL Editor
4. 执行 [`schema.sql`](/Users/kanyeast/Documents/扑克账本设计/记账网站/supabase/schema.sql)
5. 在 `Project Settings -> API Keys` 里复制：
   - `Project URL`
   - `Publishable key`
6. 在 SQL Editor 再执行一段成员初始化 SQL，例如：

```sql
insert into public.workspace_members (workspace_id, email, display_name, role)
values
  ('modern-casino', 'you@example.com', '你自己', 'owner'),
  ('modern-casino', 'partner@example.com', '合伙人', 'editor');
```

## 2. 写入前端配置

编辑 [`config.js`](/Users/kanyeast/Documents/扑克账本设计/记账网站/apps/web/src/config.js)，填入：

```js
window.POKER_LEDGER_CONFIG = {
  syncProvider: "supabase",
  supabaseUrl: "https://YOUR_PROJECT_REF.supabase.co",
  supabaseAnonKey: "YOUR_SUPABASE_PUBLISHABLE_KEY",
  supabaseTable: "workspace_state",
  shareTable: "session_shares",
  workspaceMembersTable: "workspace_members",
  workspaceId: "modern-casino",
  siteTitle: "开局账本",
};
```

说明：

- `workspaceId` 相当于共享账本房间号
- 只有加入 `workspace_members` 的邮箱登录后，才能访问完整账本
- 外部合作伙伴只能通过你生成的单账单分享链接查看对应那一场
- 如果你想区分不同牌局团队，可以复制一份站点或者改不同的 `workspaceId`

## 3. 发布到 GitHub Pages

仓库已经带了 GitHub Pages 工作流：

- 工作流文件：[deploy-pages.yml](/Users/kanyeast/Documents/扑克账本设计/.github/workflows/deploy-pages.yml)
- 发布目录：`记账网站/apps/web`

只要仓库推到 GitHub 并启用 Pages，后续每次推送到 `main` 都会自动发布。

## 4. 权限说明

当前方案是“公开前端 + 受控数据权限”，特点是：

- 优点：Pages 可以继续公开托管，但完整账本不会对访客开放
- 优点：你可以按成员邮箱控制谁能看全部账单和英雄榜
- 优点：你可以给单场牌局生成受限分享链接
- 限制：前端代码仍然是公开的，真正安全性依赖 Supabase RLS

这意味着：

- 不在成员表里的邮箱，即使登录，也不能看完整账本
- 分享链接默认能看一场账单，当前实现有效期是 14 天
- 如果分享链接被转发，拿到链接的人都能看那一场账单

## 5. 现在这套包装适合什么场景

适合：

- 你和固定合伙人共同记账
- 同时需要把某一场账单单独发给外部合作伙伴
- 希望尽快上线，但又不想把全部账本直接公开

不适合：

- 完全面向公开用户
- 需要非常细的组织级权限和审计
- 需要复杂多人并发合并而不是最后写入覆盖
